# 执行规则

编写 SQL 查询时：

## 通用规则
- 对于复杂查询，始终先使用 `sql_validate` 验证后再执行 `sql_query`
- 始终包含 LIMIT 子句（默认 LIMIT 1000）
- 使用明确的列名，避免使用 SELECT *
- 为计算列使用有意义的别名
- 正确处理 NULL 值（使用 COALESCE、NULLIF、IFNULL 等）
- 进行除法运算时，始终防止除以零的情况
- 使用目标数据库引擎对应的日期/时间函数
- 比较日期时，明确处理时区问题

## 查询错误恢复
- 如果查询返回错误，仔细阅读错误信息
- 常见问题：列未找到、类型不匹配、语法错误、列名歧义
- 自行修复 SQL 并重试，不要让用户来修复

## 性能优化
- 使用 WHERE 子句尽早过滤数据
- 当 JOIN 足够时，避免不必要的子查询
- 对于大表，先用较小的 LIMIT 验证结果，再运行完整查询

# 审查规则

获取查询结果后，按以下格式提供分析：

## 分析报告
1. **摘要**：用一句话回答用户的问题
2. **关键发现**：以要点形式列出重要的数据发现
3. **数据详情**：展示相关数据并提供上下文
4. **查询方法**：简要描述查询思路
5. **注意事项**：数据质量问题、假设条件或局限性

## 格式要求
- 使用表格展示结构化的数据对比
- 合理四舍五入小数位（货币：2 位小数，百分比：1 位小数）
- 包含单位（货币符号、% 等）
- 对于时间序列数据，注明时间范围和粒度
