# Plan Rules

理解用户问题并制定分析计划的阶段。此阶段不涉及 SQL 编写，专注于分析思路。

## 1. 理解用户问题

- 识别核心诉求：用户想要什么指标或答案？
- 提取维度：结果需要按什么分组或拆解（如按日期、类别、地区）？
- 提取过滤条件：什么条件限制了数据范围（如日期范围、状态、特定 ID）？
- 识别排序/排名需求：用户是否需要 Top N、Bottom N 或特定排序？
- 确定输出形式：单个数字、列表、对比表、时间趋势？
- 如果问题有歧义，优先做合理假设并说明，避免过多追问。

## 2. 发现相关表

- 使用 `sql_list_tables` 查看所有可用表（如果当前对话中尚未了解）。
- 根据用户问题判断哪些表可能相关。
- 注意命名模式暗示的关系（如多个表中出现 `user_id`）。

## 3. 检查 Schema

- 对每个相关表使用 `sql_get_schema`，了解：
  - 列名和数据类型
  - 主键和外键
  - 可为空的列（影响 JOIN 和聚合行为）
  - 索引信息（对性能敏感的查询有参考价值）
- 绝不猜测列名 — 必须通过 schema 检查确认。
- 注意区分数据类型：`INT`、`DECIMAL`、`VARCHAR`、`DATE`、`DATETIME`、`TIMESTAMP`、`ENUM` 等。

## 4. 制定分析计划

在写 SQL 之前，先规划分析思路：

### 简单问题
- 直接确定：需要哪些表、哪些列、什么过滤条件、什么聚合方式。
- 一步到位，无需拆分。

### 复杂问题需要拆解
识别以下场景并拆分为多个步骤：

- **跨时间段对比**：需要分别计算两个时间段的指标再对比。
- **百分比/占比计算**：分子和分母可能来自不同的分组。
- **多表关联（3+ 张表）**：先理清表之间的关联路径。
- **漏斗分析/队列分析**：需要逐步过滤的顺序逻辑。
- **排名后再过滤**：需要先排名再取子集。

### 拆解步骤模板
- **Step A — 验证假设**：先跑一个小的探索性查询（如 `SELECT DISTINCT status FROM orders LIMIT 100`）来了解数据的实际值。
- **Step B — 确认关联关系**：验证表之间的 JOIN 路径是否正确，是否会产生重复行。
- **Step C — 制定最终查询计划**：明确最终查询的结构（单查询 / CTE / 子查询），确定聚合层级。

## 5. 预判边界情况

在制定计划时提前考虑：

- **NULL 值**：JOIN 键中的 NULL 是否会导致丢行？聚合列中的 NULL 是否会影响结果？
- **重复数据**：JOIN 是否会产生重复行导致 COUNT 或 SUM 膨胀？
- **空数据**：如果过滤后没有匹配的行，查询是否仍能正常运行？
- **日期边界**：过滤条件是包含还是排除边界？用户说的"上个月"具体指哪个范围？
- **时区**：时间戳是 UTC 存储还是本地时间？用户期望的结果是哪个时区？
- **数据量**：表可能有多大？是否需要先用小 LIMIT 验证？

## 6. 输出计划

计划完成后，在脑中（或对用户简要说明）明确以下内容后再进入 SQL 编写阶段：

1. 需要查询哪些表
2. 表之间的关联方式
3. 过滤条件是什么
4. 聚合/分组方式是什么
5. 排序和限制条件
6. 是否需要分步查询
7. 需要注意的边界情况
